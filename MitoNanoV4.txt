MitoNano Engine V4 — Exascale-Ready, Sparse-Voxel Nanomedicine Platform

This document integrates all eight tracks from Improvements_For_MitoNano.txt into the V3 architecture (MitoNanoV3), producing a coherent V4 Rust workspace. As with the V2→V3 upgrade, every V1/V2/V3 module is preserved and extended, not replaced — existing params.json files keep deserializing (new fields carry #[serde(default)]), existing CLI invocations keep working, and every new subsystem is additive and feature-gated behind a new Cargo feature.

Where V3's theme was biological fidelity (hemodynamics, kinetics, swarm cognition), V4's theme is computational sparsity, numerical precision management, and differentiable architecture — the three levers needed to go from "high-performance workstation tool" to "exascale-ready simulation platform."

4. src/sim/sparse_lbm.rs — NEW: Sparse-Voxel Lattice Boltzmann (Improvement #1)

Replaces the dense N × N × N allocation in lbm.rs with an indirect-addressing scheme: only cells flagged fluid (not solid tissue) get a slot in a contiguous distribution-function buffer. An index_map (dense grid coordinate → sparse slot, or u32::MAX for "not present") makes streaming a single indexed gather/scatter instead of a full-volume sweep. Since biological vasculature typically occupies under 10% of its bounding box, this is a 70–90% VRAM reduction, and — just as importantly — a proportional reduction in work, since collide/stream no longer visits solid cells at all.

sparse_lbm.rs is written as a drop-in alternative backend: Environment holds either a LbmField (V2/V3 dense) or a SparseLbmField (V4), selected by Params::lbm_storage, and ParticulateField (V3, hemodynamics.rs) is generic over either backend via the shared FluidBackend trait introduced here.

#![cfg(feature = "sparse-lbm")] use roaring::RoaringBitmap; /// Shared interface `hemodynamics.rs` and `environment.rs` code against, so /// V3 code that took `&LbmField` can be ported to accept `&dyn FluidBackend` /// with no behavioral change when `lbm_storage == Dense`. pub trait FluidBackend { fn dims(&self) -> [usize; 3]; fn is_solid(&self, x: usize, y: usize, z: usize) -> bool; fn macroscopic(&self, x: usize, y: usize, z: usize) -> (f64, [f64; 3]); fn step(&mut self, tau: f64, inlet_velocity: [f64; 3]); } /// D3Q19 lattice, but distribution functions live in a compacted buffer /// indexed via `index_map`, not a dense `dims[0]*dims[1]*dims[2]*19` array. pub struct SparseLbmField { pub dims: [usize; 3], /// dense-coordinate -> sparse-slot; u32::MAX sentinel = solid/absent. index_map: Vec<u32>, /// sparse-slot -> dense coordinate, for writing results back out. slot_coords: Vec<[u32; 3]>, /// f[slot * 19 + q] — only occupied (fluid) cells have a slot. f: Vec<f64>, f_scratch: Vec<f64>, n_fluid_cells: usize, } const Q: usize = 19; // D3Q19 lattice velocity set + weights, identical to lbm.rs — reused rather // than duplicated in the real crate via `pub(crate)` constants in lbm.rs. use crate::sim::lbm::{D3Q19_VELOCITIES, D3Q19_WEIGHTS}; impl SparseLbmField { /// Builds the index map from an existing solid mask (e.g. produced by /// `mesh_import`/`AnatomyBvh::signed_distance`, same input V2/V3 dense /// LBM used). `occupancy_hint` (Params::occupancy_hint) is only a /// `Vec::with_capacity` sizing hint — correctness does not depend on it. pub fn from_solid_mask(dims: [usize; 3], solid: &[bool], occupancy_hint: f64) -> Self { let n = dims[0] * dims[1] * dims[2]; let est_fluid = ((n as f64) * (1.0 - occupancy_hint)).ceil() as usize; let mut index_map = vec![u32::MAX; n]; let mut slot_coords = Vec::with_capacity(est_fluid); let (dx, dy, _dz) = (dims[0], dims[1], dims[2]); for i in 0..n { if !solid[i] { let z = i / (dx * dy); let y = (i / dx) % dy; let x = i % dx; index_map[i] = slot_coords.len() as u32; slot_coords.push([x as u32, y as u32, z as u32]); } } let n_fluid_cells = slot_coords.len(); let f = vec![1.0 / Q as f64; n_fluid_cells * Q]; // equilibrium at rest let f_scratch = f.clone(); SparseLbmField { dims, index_map, slot_coords, f, f_scratch, n_fluid_cells } } #[inline] fn dense_idx(&self, x: usize, y: usize, z: usize) -> usize { x + self.dims[0] * (y + self.dims[1] * z) } #[inline] fn slot_of(&self, x: usize, y: usize, z: usize) -> Option<usize> { let s = self.index_map[self.dense_idx(x, y, z)]; if s == u32::MAX { None } else { Some(s as usize) } } /// One BGK collide + indirect-addressed stream pass. The key difference /// from `LbmField::step`: the outer loop is `0..n_fluid_cells`, not /// `0..dims[0]*dims[1]*dims[2]` — solid cells are never visited, never /// allocated, and never touch the memory bus. rayon parallelizes over /// `slot_coords` exactly as V2/V3 parallelized over the dense grid. pub fn step(&mut self, tau: f64, inlet_velocity: [f64; 3]) { use rayon::prelude::*; let slot_coords = &self.slot_coords; let index_map = &self.index_map; let dims = self.dims; let f_in = &self.f; let f_out = &mut self.f_scratch; f_out.par_chunks_mut(Q).enumerate().for_each(|(slot, f_cell_out)| { let [x, y, z] = slot_coords[slot]; let (x, y, z) = (x as usize, y as usize, z as usize); // Macroscopic moments (rho, u) from this cell's distributions. let f_cell_in = &f_in[slot * Q..slot * Q + Q]; let rho: f64 = f_cell_in.iter().sum(); let mut u = [0.0; 3]; for q in 0..Q { for a in 0..3 { u[a] += f_cell_in[q] * D3Q19_VELOCITIES[q][a]; } } for a in 0..3 { u[a] /= rho.max(1e-12); } // BGK collide (identical physics to lbm.rs's dense path). let mut f_post = [0.0; Q]; for q in 0..Q { let eu: f64 = (0..3).map(|a| D3Q19_VELOCITIES[q][a] * u[a]).sum(); let uu: f64 = (0..3).map(|a| u[a] * u[a]).sum(); let feq = D3Q19_WEIGHTS[q] * rho * (1.0 + 3.0 * eu + 4.5 * eu * eu - 1.5 * uu); f_post[q] = f_cell_in[q] - (f_cell_in[q] - feq) / tau; } // Indirect-addressed stream: for each lattice direction, look up // the neighbor's *sparse slot* via index_map (O(1)); if the // neighbor is solid (sentinel), bounce-back in place instead of // writing off into nonexistent memory — this is the entire // trick that makes sparse storage correct without a dense scan. for q in 0..Q { let dir = D3Q19_VELOCITIES[q]; let nx = x as isize + dir[0] as isize; let ny = y as isize + dir[1] as isize; let nz = z as isize + dir[2] as isize; let in_bounds = nx >= 0 && ny >= 0 && nz >= 0 && (nx as usize) < dims[0] && (ny as usize) < dims[1] && (nz as usize) < dims[2]; if in_bounds { let n_dense = (nx as usize) + dims[0] * ((ny as usize) + dims[1] * (nz as usize)); if index_map[n_dense] != u32::MAX { // Streamed value lands in *this* cell's output slot for // the opposite direction — matches lbm.rs's push-stream // convention, just gated by the index map instead of a // dense bounds/solid check. f_cell_out[q] = f_post[q]; continue; } } // Solid or domain-boundary neighbor: bounce-back. f_cell_out[opposite_direction(q)] = f_post[q]; } }); std::mem::swap(&mut self.f, &mut self.f_scratch); apply_inlet_velocity(self, inlet_velocity); 5. gpu/spatial_hash.rs + shaders/spatial_hash.wgsl — NEW: GPU Spatial Hashing (Improvement #2)

V3's SwarmGraph::build (in sim/gnn.rs) rebuilds a k-NN graph on the CPU every tick via an O(n²) pairwise-distance-then-sort scan (fine for thousands of agents, prohibitive for millions/billions). This module replaces that neighbor-discovery step — not the message-passing itself — with a GPU bucket-sort hash grid: agents are assigned a bucket ID from their position, a parallel sort groups agents by bucket, and neighbor queries become a bounded lookup over the (typically 27) buckets adjacent to an agent's own, each an O(1)-amortized memory read instead of a tree descent.

7. sim/autodiff.rs — NEW: Differentiable Simulation, "Grad-Tune" (Improvement #4)

V3's AutoTuner (sim/calibration.rs) treats the simulation as a black box: it perturbs Params, re-runs the whole simulation, and scores the result — an evolutionary strategy needing hundreds to thousands of full runs to converge on 100+ tunable variables. autodiff.rs instead makes the core kernels mathematically differentiable via forward-mode dual numbers (num_dual::Dual64), so the engine can compute ∂error/∂param directly and calibrate with gradient descent in 10–20 iterations.

#![cfg(feature = "autodiff")] use num_dual::{Dual64, DualNum}; use crate::sim::params::Params; /// A tunable parameter, generalized from `calibration.rs`'s `TunableParam` /// (V3) to carry a dual-number representation so its value can flow through /// differentiable kernel re-implementations and accumulate a gradient. #[derive(Clone, Debug)] pub struct GradTunable { pub name: String, pub value: Dual64, // .re = current value, .eps = accumulated d(error)/d(value) pub lower_bound: f64, pub upper_bound: f64, } /// Differentiable re-implementation of the subset of kernels that actually /// feed the calibration targets (opsonization rate, zeta-potential-driven /// clearance, ligand-receptor Ka/Kd occupancy) — NOT a full differentiable /// LBM/GNN engine. Takes the same /// `LabObservation` targets format `AutoTuner::from_targets_file` (V3) /// already reads, so existing lab-data JSON files work unchanged. pub struct GradTuner { pub tunables: Vec<GradTunable>, pub targets: Vec<crate::sim::calibration::LabObservation>, pub lr: f64, pub max_iters: u32, } impl GradTuner { pub fn from_targets_file(path: &str, params: &Params) -> anyhow::Result<Self> { let targets = crate::sim::calibration::load_targets(path)?; for _iter in 0..self.max_iters { let mut total_loss = Dual64::from(0.0); for obs in &self.targets { let predicted = DifferentiableModel::occupancy( self.tunable_by_name("ligand_density_surface"), self.tunable_by_name("ligand_ka"), self.tunable_by_name("ligand_kd_off"), self.tunable_by_name("receptor_density"), /* shear_stress for this observation */ 1.0, ); let err = predicted - Dual64::from(obs.observed_fraction_reaching_target); total_loss = total_loss + err * err; } loss_history.push(total_loss.re); for t in self.tunables.iter_mut() { let grad = total_loss.eps; // d(total_loss)/d(this tunable), since it was seeded via .derivative() let updated = (t.value.re - self.lr * grad).clamp(t.lower_bound, t.upper_bound); t.value = Dual64::from(updated); } } GradTuneResult { final_values: self.tunables.iter().map(|t| (t.name.clone(), t.value.re)).collect(), loss_history, } } fn tunable_by_name(&self, name: &str) -> Dual64 { self.tunables.iter().find(|t| t.name == name).map(|t| t.value) .unwrap_or(Dual64::from(0.0)) } } #[derive(Clone, Debug)] pub struct GradTuneResult { pub final_values: Vec<(String, f64)>, pub loss_history: Vec<f64>, // exposed to viz/plot.rs for a convergence-curve plot }

8. gpu/multi_gpu.rs — NEW: Multi-GPU Domain Decomposition (Improvement #5)

For full-body biodistribution runs, a single GPU's VRAM is a hard ceiling even with sparse storage (Improvement #1) and mixed precision (Improvement #3). This module splits the voxel grid into contiguous sub-domains — one per GPU — and uses NCCL for the halo-exchange communication LBM streaming needs across sub-domain boundaries every tick.

#![cfg(feature = "multi-gpu")] use anyhow::{Context, Result}; use nccl_sys::*; use std::os::raw::c_void; /// One GPU's slice of the global voxel grid, plus its NCCL communicator /// handle and the halo buffers used to exchange boundary-layer distribution /// functions with its neighbors every tick. pub struct GpuDomain { pub device_index: i32, pub local_dims: [usize; Wiring into main.rs / Simulation: when Params::num_gpus > 1 (requires --features multi-gpu), Simulation::new constructs a MultiGpuSim instead of a single GpuResidentSim, and Simulation::step delegates to it exactly as V3's single-GPU gpu_resident path did. 9. sim/engine.rs — Updated: Temporal Sub-sampling / Multi-Rate Integration (Improvement #6)

V1–V3's Simulation::step updated every subsystem — fluid field, GNN swarm policy, immune response, Behavior Trees, SDE integration — on every tick. Fluid flow, protein-corona formation, and "docking" decisions genuinely happen at different timescales; forcing the slow ones to re-evaluate every tick wastes the most expensive part of the agent loop (GNN message-passing) on ticks where nothing meaningful has changed since the last evaluation.

10. gpu/bt_jit.rs — NEW: JIT-Compiled Behavior Trees (Improvement #7)

V3's behavior_tree.wgsl hardcodes the one Behavior Tree actually used by the default scenarios directly into the kernel, explicitly leaving "a data-driven tree interpreter" as a V4 candidate. bt_jit.rs delivers that: it walks the CPU-side BTNode tree (sim/behavior_tree.rs, unchanged) and generates a specialized WGSL compute shader with the tree's specific checks compiled in as straight-line code — no runtime branching table, no if/else chain interpreting a generic tree structure on the GPU, just the user's exact tree flattened into a kernel the Warp/Wavefront scheduler can run at full occupancy.

16. Suggested Next Steps (V5 candidates)

Extend SparseLbmField from a static index map to a dynamically-updated one for scenarios where vasculature itself remodels (angiogenesis) mid-run, rather than assuming a fixed solid/fluid mask for the whole simulation.

Port GnnPolicy::act's message-passing matmuls fully onto the GPU (currently CPU-side ndarray even when use_gpu_spatial_hash supplies GPU-discovered neighbors), closing the loop so billion-agent swarms never round-trip agent features through the CPU at all.

Extend GradTuner from forward-mode dual numbers (adequate for the ~10 parameters in the current DifferentiableModel) to a proper reverse-mode tape once the differentiable kernel set grows to the "100+ variables" scale the improvements doc's Grad-Tune section envisions — forward-mode's per-parameter cost stops being negligible well before then.

Extend multi_gpu.rs's 1D slab decomposition to a proper 3D domain decomposition (octree-based, ideally sharing partitioning logic with AnatomyBvh) for organ geometries that aren't well-approximated by slicing along a single axis.

Extend BtJitCompiler to Selector-node short-circuiting with per-branch WGSL control flow (the current sketch elides this for brevity) and add a small test harness comparing JIT-compiled kernel output against the CPU BehaviorTree evaluator tick-for-tick, so tree changes can't silently diverge between the two backends.

Wire Params::precision_mode into sim/autodiff.rs so Grad-Tune's own kernel evaluations can optionally run mixed-precision too, rather than always at f64 via Dual64.
