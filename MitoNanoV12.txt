MitoNano Engine V12 — From Neighbor-Coupled, Factor-Correlated, Regime-Targeted, Trajectory-Conditioned, Branch-Fair, Catalog-Adaptive, Federated-Authority Clinical Infrastructure to Boundary-Attenuated, Rank-Adaptive, Transition-Calibrated, Sequence-Encoded, Audit-Verified, Cluster-Corroborated, Risk-Proportional Clinical Infrastructure

This document integrates the seven "Suggested Next Steps (V12 candidates)" from MitoNanoV11.txt §11 into the V11 architecture, producing a coherent V12 Rust workspace. As with every prior major version, all V1–V11 modules are preserved and extended, not replaced — existing params.json files keep deserializing (new fields carry #[serde(default)]), existing CLI invocations keep working, and every new subsystem is additive and feature-gated behind a new Cargo feature that layers on top of its V11 parent feature rather than replacing it.

Where V11's theme was closing the gap between "the model is locally correct" and "the model is correct in context," V12's theme is closing the gap between "the model is context-aware" and "the model knows how much to trust that context": a diffusion coupling term no longer treats every element boundary as equally permeable, a shared-factor rank is no longer a config-time guess, a regime-target confidence floor no longer applies one number to transition types with very different predictability, a trajectory feature no longer collapses a state history into a quadratic fit, a fairness class is no longer a hand-declared label that can silently go stale, a catalog hot-swap is no longer decided in isolation from what every other track is doing, and a federated-authority quorum is no longer blind to the size of what it's being asked to approve.

3. Pillar #1 — Boundary-Attenuated Fiber Coupling (sim/morphogenesis.rs)

Problem it closes: DiffusionCoupledRemodelingModel (V11) couples every pair of face-adjacent elements with a weight derived purely from shared-face area. Real tissue is not a single mechanically uniform continuum — fascicle boundaries (endotenon/perimysium-like septa) exist precisely to let bundles reorient somewhat independently of their neighbors across the boundary. Uniform face-adjacency coupling smooths across those boundaries just as eagerly as it smooths within a fascicle, erasing a real anatomical discontinuity the V11 model has no way to represent.

Fix: derive a per-edge attenuation factor from a TissueBoundaryMap (a segmentation, precomputed offline or supplied alongside the mesh) and multiply it into NeighborFiberCouplingGraph's edge weights at construction time, rather than changing the diffusion step itself. Setting the boundary map to "no boundaries" (attenuation ≡ 1.0 everywhere) recovers V11 exactly.

4. Pillar #2 — Rank-Adaptive Shared-Factor Correlation (gpu/adjoint_gpu.rs, sim/adjoint.rs)

Problem it closes: SharedFactorCorrelationEstimator (V11) fixes the factor count k at construction time. A long-running fleet's number of genuinely independent shock sources (distinct cooling zones, distinct power circuits, distinct firmware cohorts) is not static — new hardware added to a facility can introduce a new independent shock source, and a facility consolidation can remove one — so a k picked correctly at deploy time silently becomes wrong months later, either underfitting (a real second shock source gets folded into the first factor's loading, muddying devices_with_high_loading) or overfitting (a redundant factor absorbs pure noise, diluting dominant_factor_variance_share).

Fix: track the incremental-SVD eigenvalue spectrum online and grow or shrink k when the gap between the smallest retained eigenvalue and the largest discarded one crosses a threshold, reusing the existing Oja's-rule loading matrix (resized, not rebuilt) so no history replay is needed.

5. Pillar #3 — Transition-Calibrated Regime Targeting (sim/rl_agent.rs)

Problem it closes: RegimeTargetedTemporalAttentionGroupingNet (V11) gates its volatility-head output behind one static target_volatility_confidence_floor. But not all regime transitions are equally predictable in target — a "load-shed → recovery" transition has a narrow, well-characterized target-volatility distribution, while a "recovery → unknown-external-shock" transition genuinely has a wide one. A single floor either falls back to V10-style extrapolation too often on the easy transitions (giving up accuracy it could have had) or trusts the volatility head too readily on the hard ones (the exact failure V11 built the floor to prevent).

Fix: replace the single floor with a per-transition-type calibrated confidence table, keyed by a coarse transition-type classifier already implicit in the base network's shift-probability head's input regime label, and calibrated offline (e.g. via temperature scaling or isotonic regression against historical target-volatility error).

6. Pillar #4 — Sequence-Encoded Trajectory Frontier (orchestration/cost_model.rs, orchestration/multi_gpu.rs)

7. Pillar #5 — Audit-Verified Per-Branch Liveness (gpu/shader_verifier.rs)

Problem it closes: BranchFairnessAnnotations (V11) are hand-declared at dispatch-graph construction time and never re-checked against reality. 8. Pillar #6 — Cluster-Corroborated Catalog Swap (sim/ivis.rs)

Problem it closes: CatalogAdaptiveImmMultiFrameTracker (V11) evaluates each track's swap candidacy independently. When a genuine dataset-wide kinetic regime change occurs, many tracks' currently-loaded hypotheses degrade together and many tracks' retrospective likelihoods favor the same catalog entry around the same tick — but V11's per-track swap_in_margin and min_ticks_between_swaps treat this exactly like uncorrelated track-level noise, which is both slower to react (each track independently needs to clear the same margin) and not distinguishable, from the swap log alone, from actual noise.

Fix: add a cluster-level aggregator that watches the rate at which independent tracks are swapping toward the same catalog entry within a short window; once that rate itself is statistically surprising (more corroborating swaps than count as coincidence under a null model), treat it as evidence of a genuine regime change and temporarily relax swap_in_margin and min_ticks_between_swaps fleet-wide for the corroborated entry, reverting once the corroborated swap rate subsides.

9. Pillar #7 — Risk-Proportional Federated Quorum (sim/provenance.rs)

Problem it closes: FederatedAuthoritySet (V11) applies one static required_cosignatures threshold to every below-floor removal, regardless of size. Fix: replace the static required_cosignatures with a magnitude-indexed threshold curve, keeping the counting/verification mechanism (AuthorityCosignature::verify, dedup-by-pubkey) entirely unchanged — only which count is required changes, and it is computed once per removal request from the request's own declared magnitude, not adjustable after the fact.

11. Suggested Next Steps (V13 candidates)

Fiber coupling: TissueBoundaryMap segments are currently static for a run; a boundary map that itself updates as remodeling proceeds (fascicle boundaries can shift under sustained load) would need the segmentation to be re-derived periodically from the evolving stress/strain field rather than fixed at mesh load.

Shared-factor correlation: rank adaptation currently only grows or shrinks by one factor at a time on a fixed cadence gate; a burst-aware variant that can jump rank by more than one in a single resize when the unexplained-residual-energy signal itself spikes sharply (a sudden multi-source shock, not a gradual fleet change) would react faster to abrupt topology changes without loosening the stability gate for the common gradual case.

Regime targeting: TransitionType is currently a fixed, hand-enumerated small set; letting the classifier head propose new transition-type clusters from unsupervised structure in encodings that persistently fall into UnknownExternalShock (and routing those through a provisional, conservatively-floored calibration entry pending offline review) would close the gap for transition types the original taxonomy didn't anticipate.

Trajectory frontier: RegionSelectionTable is currently a static offline-fit lookup; an online drift detector comparing live encoder-vs-quadratic residuals against the table's assumptions (flagging when a region's prefer_encoder decision has stopped matching realized outcomes) would catch the table going stale the same way Pillar #5 catches stale fairness-class annotations.

Per-branch liveness: the audit currently only flags a mismatch; a bounded auto-remediation mode — narrowing (never widening) the enforced bound toward the stricter of the declared and observed class's bound when a mismatch persists past a configurable review window, with the change itself logged and reversible — would shrink the exposure window between a policy change and a human noticing the audit log.

Catalog-adaptive IMM: corroboration is currently evaluated within a single flat time window; a hierarchical corroboration check (does the rate of corroboration events itself look anomalous relative to the dataset's own historical corroboration-rate baseline, not just a fixed track-count threshold) would let the trigger adapt to datasets with naturally noisier or quieter track populations.

Federated authority: the risk-proportional curve is currently keyed only on removal magnitude; incorporating a second axis — how recently the same authority subset last cosigned a below-floor removal — into the required-count lookup would let the quorum curve raise friction on repeated high-magnitude removals clustered in time, not just on any single request's size in isolation.
