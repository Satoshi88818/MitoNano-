MitoNano Engine V10 — From Heterogeneous-Law, Predictive, Continuously-Parameterized, Dynamically-Proven, Multi-Model, Quorum-Governed Clinical Infrastructure to Self-Remodeling, Correlation-Aware, Anticipatory, State-Conditioned, Liveness-Proven, Kinetically-Grounded, Dual-Authority Clinical Infrastructure

This document integrates the seven "Suggested Next Steps (V10 candidates)" from MitoNanoV9.txt §14 into the V9 architecture, producing a coherent V10 Rust workspace. As with every prior major version, all V1–V9 modules are preserved and extended, not replaced — existing params.json files keep deserializing (new fields carry #[serde(default)]), existing CLI invocations keep working, and every new subsystem is additive and feature-gated behind a new Cargo feature that layers on top of its V9 parent feature rather than replacing it.

Where V9's theme was heterogeneity-within-heterogeneity and foresight, V10's theme is closing the remaining gap between "the model reacts correctly" and "the model itself is honest about its own drift": a fiber direction is no longer frozen at mesh load, a throughput forecast is no longer computed as if devices fail independently, a window-length controller no longer only reacts to volatility it has already seen, a cost/latency frontier no longer ignores the live state of the system it is scheduling onto, a dispatch-graph proof no longer stops at "every reachable branch is individually safe" without also proving the scheduler must eventually reach one, a particle tracker's hypothesis set is no longer fixed at two hand-picked motion models, and a notary quorum is no longer, by itself, sufficient protection against a majority-coercion scenario.

4. src/sim/morphogenesis.rs — EXTENDED: Remodeling Fiber Architecture (V10 Pillar #1)

V9's AnisotropicHyperelasticModel reads a FiberStructureTensor computed once when the mesh is loaded and never updates it thereafter — correct for a short simulation window, but wrong for exactly the fibrotic-progression and regeneration timescales V9's Oncology preset targets, where real collagen fiber orientation itself reorganizes toward the direction of sustained mechanical loading. RemodelingAnisotropicHyperelasticModel keeps V9's fixed-tensor branch as the default and adds a co-evolving branch that integrates a small reorientation ODE alongside the existing per-element finite-strain solve.

/// V10: per-element fiber remodeling state. Composes V9's /// `FiberStructureTensor` as its current value and adds the machinery /// to advance it. `RemodelingAnisotropicHyperelasticModel` never /// mutates a region's tensor directly under `FiberRemodelingMode:: /// FixedAtLoad` — this struct is only constructed and advanced when /// `CoEvolving` is active, so a `FixedAtLoad` run is byte-identical to /// V9's `AnisotropicHyperelasticModel`. pub struct FiberRemodelingState { /// V9's structure tensor, reused as the *current* fiber direction. pub current: FiberStructureTensor, /// Exponential-moving-average estimate of the dominant stress /// eigenvector direction this element has experienced recently — /// the "target" the fiber direction chases, not a snapshot of the /// instantaneous stress (which would make remodeling as noisy as /// the underlying elastodynamics, defeating the point of a /// progression-timescale process). stress_direction_ema: nalgebra::Vector3<f64>, ticks_since_last_reorientation_step: u64, } impl FiberRemodelingState { pub fn from_fixed(initial: FiberStructureTensor) -> Self { Self { stress_direction_ema: initial.principal_direction(), current: initial, ticks_since_last_reorientation_step: 0, } } /// Advances the EMA with this tick's Cauchy stress principal /// direction, and — once `fiber_remodeling_time_constant_ticks` /// have accumulated — rotates `current` a bounded step toward the /// EMA, scaled by `fiber_remodeling_stress_sensitivity`. The step /// is bounded (never a full snap-to-target) because real collagen /// reorganization is itself gradual; an unbounded step would let /// the anisotropic law's *own* stress output whipsaw the fiber /// direction it depends on, an unstable feedback loop V10 /// deliberately avoids by keeping the ODE's characteristic /// timescale strictly slower than the elastodynamic solve's. pub fn advance(&mut self, cauchy_stress: &nalgebra::Matrix3<f64>, sensitivity: f64, time_constant_ticks: u64) { let instantaneous_principal = principal_stress_direction(cauchy_stress); let alpha = 1.0 / (time_constant_ticks as f64).max(1.0); self.stress_direction_ema = (self.stress_direction_ema * (1.0 - alpha) + instantaneous_principal * alpha).normalize(); self.ticks_since_last_reorientation_step += 1; if self.ticks_since_last_reorientation_step >= time_constant_ticks { self.current = self.current.rotated_toward(&self.stress_direction_ema, sensitivity); self.ticks_since_last_reorientation_step = 0; } } } pub struct RemodelingAnisotropicHyperelasticModel { base: AnisotropicHyperelasticModel, // V9, unchanged — still computes stress/tangent from *current* tensor remodeling_states: HashMap<ElementId, FiberRemodelingState>, // only populated for FiberReinforced elements } impl RemodelingAnisotropicHyperelasticModel { pub fn tick(&mut self, params: &Params, per_element_cauchy_stress: &HashMap<ElementId, nalgebra::Matrix3<f64>>) { if params.fiber_remodeling_mode != FiberRemodelingMode::CoEvolving { return; // V9-identical: tensors never move } for (element, state) in self.remodeling_states.iter_mut() { if let Some(stress) = per_element_cauchy_stress.get(element) { state.advance(stress, params.fiber_remodeling_stress_sensitivity, params.fiber_remodeling_time_constant_ticks); self.base.set_structure_tensor(*element, state.current); // feeds back into V9's stress/tangent term } } } } 

gpu/shaders/fem_assemble_anisotropic_remodeling.wgsl mirrors V9's fem_assemble_anisotropic.wgsl assembly kernel exactly for the stress/tangent computation itself, and adds one additional pass — run at fiber_remodeling_time_constant_ticks cadence, not every tick — that reads back each element's dominant stress eigenvector and writes the rotated structure tensor into the same buffer V9's kernel already reads from, so the assembly kernel itself needs no branch for "is this element remodeling."

5. src/sim/adjoint.rs / src/gpu/adjoint_gpu.rs — EXTENDED: Cross-Device-Correlated Predictive Rebalancing (V10 Pillar #2)

V9's ThroughputTrendForecaster fits an independent linear trend per device and migrates shards off whichever device is forecast to bottleneck first — sound when devices degrade for independent reasons, but wrong when a co-scheduled job stealing cycles, or a shared thermal envelope throttling a whole node, degrades several devices at once for the same underlying cause. A migration plan built from independent forecasts can shed load onto a device that looks fine only because its correlated degradation hasn't shown up in its own trend yet.

/// V10: online pairwise covariance of per-device throughput-forecast /// residuals (not raw throughput — residuals against each device's own /// V9 linear trend), updated incrementally over /// `adjoint_forecast_correlation_window_samples`. Residual covariance, /// not raw-value covariance, isolates *shared shock* from each /// device's already-modeled individual trend. pub struct CrossDeviceCorrelationEstimator { window: usize, residual_history: HashMap<DeviceId, VecDeque<f64>>, } impl CrossDeviceCorrelationEstimator { pub fn record_residual(&mut self, device: DeviceId, forecast: f64, observed: f64) { let hist = self.residual_history.entry(device).or_default(); hist.push_back(observed - forecast); if hist.len() > self.window { hist.pop_front(); } } /// Pearson correlation between two devices' residual series over /// the shared overlapping window. 6. src/sim/rl_agent.rs — EXTENDED: Anticipatory Window-Length Control (V10 Pillar #3)

V9's window-length controller predicts a window length from recent coupling-score variance — an improvement over V8's fixed window, but still one reclustering cadence behind a genuine volatility-regime change, exactly the gap V9 itself closed for shard rebalancing (Pillar #2) but left open here. AnticipatoryTemporalAttentionGroupingNet adds a small classifier head trained to predict the probability of an imminent regime shift, and pre-emptively triggers a reclustering when that probability crosses qmix_regime_shift_trigger_threshold, rather than waiting for the shift to show up in observed variance.

pub struct AnticipatoryTemporalAttentionGroupingNet { base: AdaptiveTemporalAttentionGroupingNet, // V9, unchanged — still the reactive window-length predictor regime_change_head: tch::nn::Sequential, // small scalar head over the same GRU encoding V9 already computes } impl AnticipatoryTemporalAttentionGroupingNet { /// Runs after V9's `predict_window_length` each tick. 7. src/orchestration/cost_model.rs / multi_gpu.rs — EXTENDED: State-Conditioned Pareto Frontier (V10 Pillar #4)

V9's ContinuousParetoFrontierModel fits one global frontier and looks it up at operator_preferred_tradeoff_weight, regardless of whether the cluster is idle or saturated — but the actually achievable cost/latency trade-off at a given weight genuinely shifts with live system state; the same weight means a different real trade-off point under light versus heavy contention. StateConditionedParetoFrontierModel folds queue depth, thermal headroom, and (optionally) a time-of-day bucket into the same regression as additional input features, so the frontier the scheduler queries is conditioned on the state at query time, not a state-blind average over all history.

/// V10: live system-state feature vector, assembled by /// `orchestration::multi_gpu` each tick. orchestration/multi_gpu.rs gains a current_state_features() accessor alongside V9's existing throughput signal history exposure; orchestration/scheduler.rs calls it once per scheduling decision and passes the result straight into query, so the scheduler itself needs no new state-tracking of its own.

8. src/gpu/shader_verifier.rs — EXTENDED: Liveness-Proven Dynamic Pipeline Verification (V10 Pillar #5)

V9's DynamicPipelineSymbolicExecutor proves that every individually reachable scheduler-selectable branch, up to max_dynamic_branch_fanout, is safe if chosen — but says nothing about whether the scheduler's own selection policy is actually guaranteed to choose one of the proved branches. A policy that could, in principle, select an unproven branch beyond the fanout bound is not ruled out by a safety-only obligation. LivenessProvingPipelineSymbolicExecutor adds a second obligation: under a stated fairness assumption, the policy must eventually reach a proved branch.

/// V10: a liveness obligation is meaningless without a fairness /// assumption (an adversarial scheduler could avoid any given branch /// forever), so V10 makes the assumption an explicit, reported input /// rather than an implicit one: the policy is assumed not to avoid /// every proved branch for more than `shader_liveness_fairness_bound_ticks` /// consecutive ticks. The obligation discharges (or is reported as /// undischarged, never silently assumed) against exactly that bound. pub struct LivenessObligation { pub fairness_bound_ticks: u64, pub proved_branch_set: Vec<DispatchGraphId>, // from V9's safety pass, unchanged } pub struct LivenessProvingPipelineSymbolicExecutor { base: DynamicPipelineSymbolicExecutor, // V9, unchanged — still runs the safety-only pass first } impl LivenessProvingPipelineSymbolicExecutor { /// Runs V9's safety pass unconditionally first — a liveness proof /// over branches that are not even individually safe is not a /// useful proof — then, only under `SafetyAndLiveness` scope, /// poses a second SMT query: does every infinite policy trace /// consistent with the fairness bound visit `proved_branch_set` /// infinitely often? A `Discharged` result and an /// `Undischarged { counterexample_trace }` result are both /// reported to `stats.rs`; V10 never treats an undischarged /// liveness obligation as equivalent to V9's safety-only proof — /// callers that need the stronger guarantee must check /// `obligation_scope` was actually honored, not just that /// `verify()` returned `Ok`. pub fn verify(&self, params: &Params, graph: &DispatchGraph) -> VerificationResult { let safety = self.base.verify(params, graph)?; // V9-identical safety obligations, unchanged if params.shader_proof_obligation_scope != ShaderProofObligationScope::SafetyAndLiveness { return Ok(safety); // V9-identical } let liveness_query = build_liveness_query( &safety.proved_branches, params.shader_liveness_fairness_bound_ticks, ); let liveness_result = self.z3_context.check_universal(liveness_query)?; Ok(safety.with_liveness(liveness_result)) } } 

9. src/sim/ivis.rs — EXTENDED: Data-Driven IMM Model Set (V10 Pillar #6)

V9's ImmMultiFrameTracker blends exactly two motion hypotheses — constant-velocity and constant-acceleration — which together still cannot represent a real binding/unbinding kinetic signature such as a sudden near-stop, a motion pattern neither smooth hypothesis fits well. DataDrivenImmMultiFrameTracker keeps V9's CV/CA pair as the always-present base hypotheses and, when a kinetic-kernel catalog is configured, adds up to ptv_imm_max_hypotheses - 2 additional hypotheses fit from sim::ivis's own historical kinetic-kernel catalog, so IMM's model probabilities can express "this looks like binding," not only "this looks accelerating."

/// V10: one additional IMM hypothesis, parameterized by a kinetic /// kernel loaded from the catalog rather than hand-derived like V9's /// CV/CA sub-filters. 10. src/sim/provenance.rs — EXTENDED: Dual-Authority Notary Governance (V10 Pillar #7)

V9's QuorumGovernedNotaryLedger requires a quorum of currently-active notaries to co-sign a rotation — a real improvement over V8's single-operator model, but a quorum drawn from the active notary set itself provides no protection if a majority of that very set is coerced or compromised simultaneously. AuthorityCosignedNotaryLedger keeps V9's notary-quorum requirement unconditionally in force and, specifically for the case of removing a notary such that the active count would fall below notary_min_active_after_rotation, additionally requires a co-signature from a designated external authority whose key is not a member of the notary set at all — the notary quorum and the external authority are both required, not offered as alternatives.

/// V10: gates only the specific action V9's quorum alone cannot /// adequately protect against — removing a notary such that the /// remaining active count drops below the configured floor. A /// rotation that adds a notary, or removes one while comfortably /// above the floor, still only needs V9's notary quorum; the external /// authority is not a universal extra hurdle, it is targeted at the /// one action a coerced notary-majority could otherwise use to push /// the active set toward a point where quorum itself becomes trivial /// to control. pub struct AuthorityCosignedNotaryLedger { inner: QuorumGovernedNotaryLedger, // V9, unchanged — notary co-signature collection untouched external_authority_pubkey: Option<VerifyingKey>, pending_authority_cosignatures: HashMap<ProposalHash, (GovernanceCosignature, u64)>, // (signature, proposed_at) } impl AuthorityCosignedNotaryLedger { fn action_requires_external_authority(&self, action: &NotaryGovernanceAction, params: &Params) -> bool { if params.notary_governance_authority_mode != NotaryGovernanceAuthorityMode::QuorumPlusExternalAuthority { return false; // V9-identical: notary quorum alone is sufficient for every action } matches!(action, NotaryGovernanceAction::Remove { .. }) && self.inner.current_notaries().len().saturating_sub(1) < params.notary_min_active_after_rotation } /// V9's `try_commit` is called only after both gates clear for an /// action requiring external authority — notary quorum (checked /// by `self.inner` exactly as in V9) AND, for this action, a /// still-fresh external co-signature. Neither gate substitutes /// for the other: a fully quorum-cosigned removal below the floor /// with no external co-signature yet still returns `None`, the /// same "no effect yet" result V9 returns for an unmet notary /// quorum, so callers observe one consistent "not yet committed" /// signal regardless of which gate is still open. pub fn try_commit(&mut self, params: &Params, proposal_hash: &ProposalHash, prev_hash: &str, index: u64) -> anyhow::Result<Option<GovernanceAnchor>> { let proposal = self.inner.pending_proposal(proposal_hash).ok_or(anyhow::anyhow!("unknown proposal"))?; if self.action_requires_external_authority(&proposal.action, params) { let Some((cosig, proposed_at)) = self.pending_authority_cosignatures.get(proposal_hash) else { return Ok(None); // notary quorum may already be met — still withheld pending external authority }; let stale = now_secs().saturating_sub(*proposed_at) > params.notary_external_authority_cosign_window_secs; if stale || !verify_cosignature(self.external_authority_pubkey.as_ref(), cosig, proposal_hash) { return Ok(None); } } self.inner.try_commit(proposal_hash, prev_hash, index) // V9-identical once both gates (or just V9's, if only one applies) clear } pub fn external_authority_cosign(&mut self, proposal_hash: ProposalHash, cosignature: GovernanceCosignature) { self.pending_authority_cosignatures.insert(proposal_hash, (cosignature, now_secs())); } }

13. Suggested Next Steps (V11 candidates)

Extend RemodelingAnisotropicHyperelasticModel from a per-element reorientation step driven purely by locally-sensed stress to one that also accounts for neighboring elements' fiber directions — real collagen remodeling is not purely local; fiber bundles mechanically couple to their neighbors' orientation through shared load paths, so a purely per-element ODE can produce a physically-implausible sawtooth fiber field across adjacent elements that a diffusion-coupled remodeling model would smooth into a coherent bundle structure.

Extend CrossDeviceCorrelationEstimator from pairwise residual correlation to a full low-rank shared-factor model — pairwise correlation catches two-device co-degradation but scales poorly and misses a genuinely cluster-wide shared shock (e.g., a facility-level cooling event) that shows up as elevated correlation across every pair simultaneously rather than any single dominant pair, which a shared-latent-factor decomposition would isolate directly instead of inferring indirectly from many pairwise coefficients.

Extend AnticipatoryTemporalAttentionGroupingNet's regime-change head from a single scalar shift-probability output to a predicted post-shift volatility level — knowing a shift is coming lets V10 trigger reclustering early, but the anticipated window length is still computed from the pre-shift encoding; a head that also predicts the target regime's characteristic volatility would let the pre-emptive reclustering choose a window length suited to the coming regime rather than an extrapolation of the one about to end.

Extend StateConditionedParetoFrontierModel from conditioning on instantaneous system state to conditioning on a short state trajectory — a queue depth of 40 that has been climbing for ten ticks and a queue depth of 40 that has been falling for ten ticks are the same instantaneous feature value but predict different achievable frontiers a few ticks out, a distinction the current point-in-time feature vector cannot represent.

Extend LivenessProvingPipelineSymbolicExecutor from a single global fairness bound to a per-branch fairness bound — some branches are inherently rarer scheduler choices by design (a fallback-only path should legitimately be selected less often than a hot path), so a single bound either falsely flags legitimate rare-but-fair branches as violations or is set loose enough to miss a genuine starvation case on a branch that should be common.

Extend DataDrivenImmMultiFrameTracker's catalog hypothesis selection from a static top-N-by-historical-match-rate ranking, fixed for the run's duration, to an online catalog hypothesis that can be swapped mid-run as a track's own recent likelihood history suggests a better-fitting kernel exists in the wider catalog than the ones currently loaded — a single dataset can contain multiple distinct kinetic regimes across different tracks or time windows, and a fixed pre-run selection cannot adapt to a regime the initial ranking under-weighted.

Extend AuthorityCosignedNotaryLedger from a single designated external authority to a configurable set of external authorities with its own quorum requirement — a single external authority is itself now a single point of coercion or compromise for exactly the below-floor-removal case V10 was designed to protect, the same structural weakness V9's move from single-operator to notary-quorum was meant to close, now recurring one layer up.
