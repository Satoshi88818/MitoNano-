**Prioritisation for implementing and testing MitoNano V14**, based on risk, clinical impact, engineering leverage, and verification feasibility.

### Prioritisation Framework
(ranked by importance):
1. **Patient/Clinical Risk** — Does this affect treatment decisions, dosing, or safety?
2. **Correctness & Safety** — How easy is it to introduce regressions or over-corrections?
3. **Impact / Leverage** — How much does it improve core system behavior?
4. **Dependencies & Complexity** — Implementation effort and interactions.
5. **Testability** — Ease of writing property tests / fuzz harnesses.

### Recommended Priority Order (V14 Implementation)

| Priority | Pillar | Reason | Recommended Approach |
|----------|--------|------|----------------------|

Bug here could cause systematic suboptimal (or unsafe) choices. | Implement first. Strong focus on property tests: "refit never worsens outcome on original window", "non-flagged regions untouched", "idempotency". |

### Testing Strategy (Layered and Rigorous)

1. **Unit + Property Tests (Highest Priority)**
   - For every pillar: Prove that `None` config = exact V13 behaviour (regression suite).
   - Invariants: "local refit only touches flagged region", "staged narrowing is independently reversible", "fuzzy quorum never lowers requirements".
   - Use `proptest` or `arbitrary` for randomised inputs (window sizes, fleet scales, noise levels).

2. **Integration Tests**
   - End-to-end simulation runs with V14 features enabled incrementally.
   - Compare traces: V13 vs V14 under identical seeds.

3. **Fuzz Harness** (Critical for #4, #5, #6)
   - Long-running simulations with injected drift, noise, and adversarial patterns.
   - Monitor for oscillations, over-corrections, or quorum bypasses.

4. **Performance & Stability**
   - Benchmark compute savings (esp. Pillar 1).
   - Stress test with growing topology (Pillar 2).

5. **Audit Logging**
   - Ensure every trace (`stats.rs`) is comprehensive enough for post-hoc review.

### Phased Rollout Plan
- **Phase 1 (MVP)**: Pillars 4 + 6 + strong property tests. This gives the highest risk reduction.
- **Phase 2**: Pillars 1 + 5 (efficiency + safety).
- **Phase 3**: Pillars 2, 7, 3.
- **Phase 4**: Full `v14-full` + long-term simulation studies.

**First-Principles Rationale**: In safety-critical systems, **correctness > optimization > new features**. Start with mechanisms that can actively *correct* or *prevent* errors (#4, #6, #5), then add those that make the system more efficient or autonomous.

This approach minimizes downside while maximizing learning from real implementation feedback — exactly as the V14 spec itself wisely suggests.
