MitoNano Engine V13 — From Boundary-Attenuated, Rank-Adaptive, Transition-Calibrated, Sequence-Encoded, Audit-Verified, Cluster-Corroborated, Risk-Proportional Clinical Infrastructure to Boundary-Adaptive, Burst-Aware, Taxonomy-Extending, Drift-Detected, Self-Remediating, Baseline-Relative, Recency-Weighted Clinical Infrastructure

This document integrates the seven "Suggested Next Steps (V13 candidates)" from MitoNanoV12.txt §11 into the V12 architecture, producing a coherent V13 Rust workspace. As with every prior major version, all V1–V12 modules are preserved and extended, not replaced — existing params.json files keep deserializing (new fields carry #[serde(default)]), existing CLI invocations keep working, and every new subsystem is additive and feature-gated behind a new Cargo feature that layers on top of its V12 parent feature rather than replacing it.

Where V12's theme was closing the gap between "the model is context-aware" and "the model knows how much to trust that context," V13's theme is closing the gap between "the model trusts context correctly right now" and "the model notices when 'right now' has quietly become 'six months ago'": a boundary map no longer assumes fascicle geometry is static for the life of a run, a rank-adaptive estimator no longer treats every shock as a one-factor-at-a-time event, a transition taxonomy no longer treats "unknown" as a permanent label, a trajectory frontier's region table no longer assumes its own offline fit stays true forever, a liveness audit no longer stops at flagging a mismatch, a catalog-swap corroboration window no longer judges a dataset's noise level against a fixed count, and a quorum curve no longer treats every below-floor request as if the same authorities hadn't just cosigned one an hour ago.

3. Pillar #1 — Re-Derived Boundary Segmentation (sim/morphogenesis.rs)

Problem it closes: TissueBoundaryMap (V12) is computed once, offline, at mesh load and never revisited. Fascicle boundaries are not geometric fixtures — under sustained load, bundles that were once mechanically distinct can fuse in behavior, and a bundle that was uniform at t=0 can develop a real internal discontinuity as remodeling proceeds. A segmentation frozen at mesh load slowly drifts out of sync with the stress/strain field it was meant to describe, and V12 has no mechanism to notice.

Fix: add an opt-in BoundarySegmentationRefreshPolicy that, on a configurable tick cadence, re-derives segment membership from the current ElementStressField using the same offline segmentation heuristic (unchanged, now runnable online), and only commits the new segmentation if it differs from the current one by more than a hysteresis threshold — preventing a boundary map that chatters between two near-equally-valid segmentations tick to tick.

// sim/morphogenesis.rs — EXTENDED /// V12, unchanged; still the boundary-aware coupling-graph builder and the /// static-map model, and V13's resegmenting model composes them as `inner`. pub use crate::sim::morphogenesis::{ BoundaryAttenuatedDiffusionCoupledRemodelingModel, TissueBoundaryMap, SegmentId, FiberCouplingBoundaryMode, NeighborFiberCouplingGraph, }; #[derive(Debug, Clone, Copy, PartialEq, serde::Serialize, serde::Deserialize)] pub enum BoundarySegmentationRefreshPolicy { Static, // V12 — map fixed at mesh load Periodic { every_ticks: u64 }, // V13 — re-derived on a fixed cadence } impl Default for BoundarySegmentationRefreshPolicy { fn default() -> Self { BoundarySegmentationRefreshPolicy::Static } // reproduces V12 exactly } /// V13: composes V12's model as `inner`. Delegates every diffusion step to /// `inner.step`, exactly as V12 did to V11's — the only new behavior is /// deciding, once per tick, whether to re-derive and hot-swap /// `inner.boundary_map` before that delegation happens. pub struct AdaptiveResegmentingBoundaryAttenuatedModel { pub inner: BoundaryAttenuatedDiffusionCoupledRemodelingModel, // V12 pub refresh_policy: BoundarySegmentationRefreshPolicy, /// Minimum fraction of elements that must change segment for a /// re-derived map to actually replace `inner.boundary_map`. Prevents /// chattering between two near-tied segmentations. pub resegmentation_hysteresis_fraction: f64, ticks_since_last_refresh: u64, } impl AdaptiveResegmentingBoundaryAttenuatedModel { pub fn step( &mut self, state: &mut FiberRemodelingState, stress_field: &ElementStressField, mesh: &FemMesh, dt_ticks: f64, ) -> DiffusionStepReport { self.ticks_since_last_refresh += 1; if let BoundarySegmentationRefreshPolicy::Periodic { every_ticks } = self.refresh_policy { if self.ticks_since_last_refresh >= every_ticks { self.ticks_since_last_refresh = 0; let candidate = TissueBoundaryMap::derive_from_stress_field( mesh, stress_field, self.inner.boundary_map.cross_segment_attenuation, ); // reuses V12's offline segmentation heuristic, now called online let changed_fraction = candidate.disagreement_fraction(&self.inner.boundary_map); if changed_fraction >= self.resegmentation_hysteresis_fraction { self.inner.inner.coupling_graph = NeighborFiberCouplingGraph::from_face_adjacency_with_boundaries(mesh, &candidate); self.inner.boundary_map = candidate; } } } // V12, unchanged: all per-tick diffusion math is inner.inner.step's. self.inner.step(state, stress_field, dt_ticks) } } 

stats.rs gains boundary_resegmentation_trace: Vec<(tick: u64, changed_fraction: f64, committed: bool)> — every candidate resegmentation attempt, whether or not it cleared the hysteresis threshold, so a run can be checked post-hoc for whether the refresh cadence was too fast (many committed: false chattering near the threshold) or too slow (large changed_fraction jumps).

Params gains fiber_coupling_boundary_refresh_policy: BoundarySegmentationRefreshPolicy (default Static, exactly reproducing V12) and fiber_coupling_resegmentation_hysteresis_fraction: f64 (default 1.0, i.e. unreachable — a params.json supplying only Static never resegments, matching V12 byte-for-byte).

4. Pillar #2 — Burst-Aware Rank Jumping (gpu/adjoint_gpu.rs, sim/adjoint.rs)

Problem it closes: RankAdaptiveSharedFactorCorrelationEstimator (V12) grows or shrinks rank by exactly one factor per cadence gate. That is the right conservative default for the common case — a fleet slowly adding one cooling zone over a quarter — but it reacts far too slowly to an abrupt multi-source shock (e.g. a facility-wide firmware rollout that simultaneously introduces three new independent behavior clusters), where the unexplained-residual-energy signal spikes sharply in a single window rather than climbing gradually.

Fix: add a burst detector that compares the current window's unexplained-residual-energy against a rolling baseline; when the spike ratio clears a configurable threshold, allow the resize step to jump rank by more than one in that single tick, bypassing the normal one-factor-per-gate cadence just for that tick. The gradual case is untouched — the burst detector is a strictly additional escape hatch, not a replacement for the existing gate.

// gpu/adjoint_gpu.rs — EXTENDED /// V12, unchanged; still does eigenvalue-gap rank selection at the normal /// one-factor-per-cadence-gate pace, and V13's estimator composes it as /// `inner`, reusing its resize machinery for both gradual and burst jumps. pub use crate::gpu::adjoint_gpu::{ RankAdaptiveSharedFactorCorrelationEstimator, SharedFactorCorrelationEstimator, AdjointForecastRankSelectionMode, }; #[derive(Debug, Clone, Copy, serde::Serialize, serde::Deserialize)] pub struct BurstDetectionConfig { /// Rolling-baseline window length (ticks) for unexplained-residual-energy. pub baseline_window_ticks: u32, /// A window's residual energy over `baseline_mean * spike_ratio_threshold` /// counts as a burst. pub spike_ratio_threshold: f64, /// Hard ceiling on how many factors a single burst jump may add, /// independent of the normal `adjoint_forecast_rank_max` clamp. pub max_factors_per_burst: u32, } /// V13: composes V12's estimator as `inner`. Every gradual, cadence-gated /// resize decision is delegated unchanged to `inner.maybe_resize`; the only /// new path is `maybe_burst_resize`, called first each tick, which can grow /// rank by more than one factor when the residual-energy signal itself /// spikes — and which the gradual path then simply sees as a fait accompli /// (its own gate no-ops for a tick where rank already moved). pub struct BurstAwareSharedFactorCorrelationEstimator { pub inner: RankAdaptiveSharedFactorCorrelationEstimator, // V12 pub burst_config: BurstDetectionConfig, residual_energy_baseline: RollingMean, // vendored rolling-stat helper, used since V9 } impl BurstAwareSharedFactorCorrelationEstimator { pub fn observe_and_maybe_resize( &mut self, window_residual_energy: f64, tick: u64, ) -> RankResizeReport { let baseline = self.residual_energy_baseline.mean(); self.residual_energy_baseline.push(window_residual_energy); let is_burst = baseline > 0.0 && window_residual_energy >= baseline * self.burst_config.spike_ratio_threshold; if is_burst { let headroom = self.inner.rank_max.saturating_sub(self.inner.current_rank()); let jump = headroom.min(self.burst_config.max_factors_per_burst); if jump > 0 { self.inner.grow_rank_by(jump); // V12's resize primitive, called with jump > 1 self.inner.reset_cadence_gate(tick); // avoid an immediate redundant gradual grow return RankResizeReport::BurstJump { factors_added: jump, trigger_ratio: window_residual_energy / baseline }; } } // V12, unchanged: normal one-factor-per-gate gradual path. self.inner.maybe_resize(window_residual_energy, tick) } } 

stats.rs gains factor_rank_burst_trace: Vec<(tick: u64, factors_added: u32, trigger_ratio: f64)>, kept alongside V12's factor_rank_history so a burst jump is visible both as a discontinuity in the rank-over-time series and as an explicit, attributable event.

Params gains adjoint_forecast_burst_detection: Option<BurstDetectionConfig> (default None, exactly reproducing V12's one-factor-per-gate-only behavior).

5. Pillar #3 — Taxonomy-Extending Transition Classifier (sim/rl_agent.rs)

Problem it closes: TransitionType (V12) is a fixed, hand-enumerated enum, and TransitionClassifierHead routes anything it can't confidently place into UnknownExternalShock, which carries the single most conservative calibration entry. If a genuinely new, recurring transition pattern emerges — one the original taxonomy's authors never saw — every instance of it is calibrated as if it were the worst-case unknown, forever, even after the pattern has repeated often enough to characterize.

Fix: track encodings that persistently classify as UnknownExternalShock; when a sub-cluster of them is dense and stable enough (measured, not guessed), propose a new provisional TransitionType variant with its own conservatively-floored calibration entry — deliberately floored below what the observed data alone would justify, and explicitly flagged for offline human review before it can be promoted to a first-class taxonomy entry.

6. Pillar #4 — Online Region-Table Drift Detection (orchestration/cost_model.rs)

Problem it closes: RegionSelectionTable (V12) is a static, offline-fit lookup deciding per-region whether the learned sequence encoder or the V11 quadratic fallback is preferred. A region's prefer_encoder decision was correct when fit, but workload composition drifts — a region that used to be encoder-favorable can become quadratic-favorable (or vice versa) — and V12 has no way to notice the table itself has gone stale, the same blind spot Pillar #5 of V12 closed for fairness-class annotations.

Fix: add an online monitor that, for every scheduling decision, retroactively compares the encoder's and the quadratic fallback's residuals against the realized outcome once it's known, accumulates a per-region running disagreement-with-table signal, and flags (never silently overrides) a region whose prefer_encoder decision has stopped matching realized outcomes often enough.

// orchestration/cost_model.rs — EXTENDED /// V12, unchanged; still the encoder-preferring frontier model and its /// static region table, and V13's monitor observes it without altering /// its routing decisions. pub use crate::orchestration::cost_model::{ SequenceEncodedParetoFrontierModel, RegionSelectionTable, LearnedTrajectoryEncoder, TrajectoryConditionedParetoFrontierModel, }; 7. Pillar #5 — Bounded Auto-Remediation for Class Mismatch (gpu/shader_verifier.rs)

Problem it closes: AuditedPerBranchFairLivenessSymbolicExecutor (V12) flags a declared-vs-observed fairness-class mismatch but takes no action — the exposure window between a policy going stale and a human reading the audit log can be arbitrarily long under low-traffic monitoring.

Fix: add an opt-in remediation mode that, only once a mismatch has persisted past a configurable review window (never on first detection), narrows — never widens — the enforced bound toward whichever of the declared or observed class's bound is stricter. Every narrowing is logged with before/after values and is explicitly reversible: a human can restore the declared bound at any time, and the mechanism only ever tightens autonomously, never loosens.

// gpu/shader_verifier.rs — EXTENDED /// V12, unchanged; still does the observed-vs-declared mismatch detection /// this pillar acts on, and V13's executor composes it as `inner`. pub use crate::gpu::shader_verifier::{ AuditedPerBranchFairLivenessSymbolicExecutor, ObservedSelectionHistory, ObservedClassThresholds, ClassMismatch, PerBranchFairLivenessSymbolicExecutor, }; #[derive(Debug, Clone, Copy, serde::Serialize, serde::Deserialize)] pub struct AutoRemediationConfig { /// Ticks a mismatch must persist, continuously, before remediation acts. pub review_window_ticks: u64, /// Safety ceiling: remediation never narrows a bound by more than this /// fraction of the declared bound in a single action. pub max_single_narrowing_fraction: f64, } #[derive(Debug, Clone, serde::Serialize, serde::Deserialize)] pub struct ReversibleBoundNarrowing { pub branch_id: BranchId, pub tick: u64, pub declared_bound_before: LivenessBound, pub enforced_bound_after: LivenessBound, pub reverted: bool, } /// V13: composes V12's executor as `inner`. 8. Pillar #6 — Baseline-Relative Corroboration Rate Anomaly (sim/ivis.rs)

Problem it closes: ClusterCorroboratedImmMultiFrameTracker (V12) evaluates corroboration within a single flat time window against a fixed track-count threshold. A dataset with naturally noisy, fast-swapping tracks and a dataset with naturally quiet, stable ones get the same trigger sensitivity, so the fixed threshold is simultaneously too loose for the noisy dataset (frequent spurious corroboration) and too tight for the quiet one (real corroboration events under-triggering).

Fix: track each dataset's own historical corroboration-event rate as a rolling baseline, and gate the cluster-swap trigger on whether the current corroboration rate is anomalous relative to that dataset's own baseline — not just whether it clears a fixed absolute count.

9. Pillar #7 — Recency-Weighted Federated Quorum (sim/provenance.rs)

11. Suggested Next Steps (V14 candidates)

Boundary resegmentation: the refresh cadence is currently a fixed tick interval; a load-adaptive cadence that shortens the interval when the resegmentation trace shows changed_fraction trending upward across successive checks (boundaries actively in motion) and lengthens it when segmentation has been stable for many checks in a row would spend the resegmentation cost where it's actually needed.

Burst-aware rank jumping: the spike-ratio threshold is currently a single global constant; a per-fleet-topology threshold that accounts for how many independent shock sources the fleet already has (a tenth source appearing is a smaller relative residual-energy jump than a second one) would keep burst sensitivity calibrated as the fleet itself grows.

Taxonomy extension: provisional clusters are currently proposed but never automatically promoted; a promotion pathway that converts a provisional cluster to a first-class TransitionType once it has accumulated enough offline-reviewed, human-confirmed instances — retiring the discounted confidence floor at that point — would close the loop the review flag currently leaves open.

Region-table drift: the monitor currently only flags; an automatic, bounded re-fit of just the flagged region's prefer_encoder decision (not the whole table) using only the realized-outcome window that triggered the flag would mirror the bounded-remediation pattern Pillar #5 established for fairness classes.

Auto-remediation: narrowing currently targets the stricter of declared/observed bounds in one step (up to the per-action ceiling); a staged narrowing that takes several smaller steps across successive persistent-mismatch windows, each independently reversible, would reduce the chance a single remediation over-corrects on a noisy one-window observation.

Corroboration baseline: the anomaly check is currently a single ratio-vs-baseline test; a two-sided variant that also flags an anomalously low corroboration rate (potential track fragmentation going undetected, not just spurious over-triggering) would give the baseline tracker symmetric coverage.

Recency-weighted quorum: the recency axis is currently keyed on exact pubkey-set match; a fuzzy-overlap variant (e.g. requiring only a majority-pubkey overlap, not an identical set) would close the gap where a coerced quorum avoids the recency penalty simply by swapping one member between requests.
